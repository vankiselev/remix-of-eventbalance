
Понял проблему: данные реально читаются с self-hosted (это видно по запросам к `superbag.eventbalance.ru`), и в ответе `/rest/v1/events` уже приходят `project_owner: "В" | "Н" | "Л"`. Значит UI показывает то, что лежит в БД, а не кэш/рендер-баг.

### Почему «ничего не поменялось»
1. Последний фикс был добавлен в `supabase/migrations/...`.
2. Ваш self-hosted CI применяет миграции **только** из `migrations/**` (см. `.github/workflows/deploy-migrations.yml`, `paths: 'migrations/**'` и цикл `for migration in migrations/*.sql`).
3. Поэтому SQL с заменой букв просто не исполнялся на self-hosted.

### План исправления

1) **Сделать правильный data-patch в нужной папке**
- Создать новый SQL-файл в `migrations/` (новое имя/таймстамп, чтобы CI не пропустил как уже применённый).
- Не использовать `supabase/migrations` для этого проекта self-hosted.

2) **Обновить все существующие значения `чей проект` в БД**
- В `public.events` заменить сокращения:
  - `н`/`Н` → `Настя`
  - `л`/`Л` → `Лера`
  - `в`/`В` → `Ваня`
- Сравнение делать через `trim(lower(project_owner))`, чтобы поймать пробелы/регистр.
- Обновить `updated_at` у затронутых строк.

3) **(Опционально, но рекомендую) Применить ту же нормализацию в `financial_transactions`**
- Там тоже есть поле `project_owner`, и «чей проект» используется в интерфейсах финансов.
- Это уберёт разные форматы в разных разделах.

4) **Защититься от повторного появления сокращений**
- В `EventsImportDialog.tsx` оставить/усилить нормализацию перед insert/update:
  - тримминг,
  - lowercase,
  - маппинг `н/л/в` в полные имена.
- Добавить единый helper (чтобы одинаково работало в импорте и в других местах, если появятся новые точки записи).

5) **Проверка после применения**
- В GitHub Actions дождаться успешного `Deploy to Self-Hosted Supabase`.
- Проверить в `/events`, что вместо `В/Н/Л` отображаются `Ваня/Настя/Лера`.
- Быстрая контрольная проверка: выборка distinct `project_owner` (должны остаться только полные имена + уже валидные другие значения).

Итог: UI корректно показывает данные из БД, но данные не были обновлены, потому что SQL не доехал до self-hosted.
